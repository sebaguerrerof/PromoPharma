# PLAN: Brand Text Bank + Smart Prompt

> **Módulo**: Mailing Composer (M6) — Extensión de "Crear con AI"  
> **Objetivo**: (1) Almacenar TODOS los textos de cada marca (manuales y AI) como un banco de referencia, (2) que la IA genere siempre basándose en ese corpus, (3) botón "Sugerir prompt" que use el banco para generar briefs inteligentes.

---

## 1. CONCEPTO GENERAL

### 1.1 Problema actual

No hay memoria de los textos que una marca produce. Cada generación AI empieza de cero:
- No sabe qué tipo de emails suele crear esta marca
- No conoce el estilo de redacción del cliente (tono, largo, vocabulario)
- No recuerda qué temas ya se han tratado
- No puede detectar qué funciona vs qué se descarta
- Los textos escritos manualmente por el cliente se pierden como referencia

### 1.2 Solución: Tres capacidades nuevas

| # | Capacidad | Descripción |
|---|-----------|-------------|
| **A** | **Brand Text Bank** | Colección Firestore `brandTextBank` que almacena TODOS los textos de la marca: subjects, bloques, prompts — tanto creados con IA como escritos manualmente. Cada entrada taggeada con tipo (informativo, científico, etc.) y características. |
| **B** | **Smart Prompt** | Botón "✨ Sugerir prompt" junto al textarea que genera un brief inteligente basándose en la marca + el banco de textos completo |
| **C** | **Corpus-Based Generation** | Cuando la IA genera un email, consulta el banco de textos de esa marca y genera **basándose en esos textos** — no genéricamente. Aprende el estilo, tono y vocabulario del cliente. |

## 2. COLECCIÓN FIRESTORE: `brandTextBank`

### 2.1 Concepto clave

**Todo texto que produce una marca se guarda** — ya sea creado con IA o escrito a mano por el cliente. Esto incluye:
- Subjects de emails
- Contenido de cada bloque (títulos, párrafos, bullets, CTAs, etc.)
- Prompts que el usuario escribió
- Textos que la IA generó

Cada entrada está **taggeada** con su tipo de email y características, para que la IA pueda filtrar y generar basándose solo en textos relevantes.

### 2.2 Estructura del documento

Cada documento = **1 mailing guardado** (contiene todos sus textos desglosados).

```typescript
// Nuevo tipo en src/types/index.ts

export type TextBankEmailType = 'promocional' | 'informativo' | 'newsletter' | 'invitación' | 'científico' | 'aviso_breve' | 'otro';
export type TextBankSource = 'manual' | 'ai' | 'ai_edited';  // ai_edited = generado por IA pero editado por el cliente

export interface BrandTextBankEntry {
  id: string;                     // auto-generated by Firestore
  brandId: string;
  brandName: string;
  tenantId: string;

  // ── Clasificación del email ──
  emailType: TextBankEmailType;   // Tipo de email (OBLIGATORIO)
  tags: string[];                 // Tags libres adicionales (ej: 'webinar', 'congreso', 'lanzamiento')
  source: TextBankSource;         // Quién lo creó: manual, IA, o IA+edición

  // ── Textos del email (lo que la IA usa como referencia) ──
  subject: string;                // Subject line del email
  preheaderText?: string;         // Preheader

  // Textos desglosados por tipo de bloque — esto es lo que la IA indexa
  texts: {
    titles: string[];             // Todos los títulos (text blocks con headingLevel)
    paragraphs: string[];         // Todos los párrafos (text blocks sin headingLevel)
    bulletPoints: string[];       // Todos los items de bullets (desglosados)
    ctaTexts: string[];           // Textos de botones CTA
    ctaLabels: string[];          // Etiquetas superiores de CTA
    heroTitles: string[];         // Títulos hero
    heroSubtitles: string[];      // Subtítulos hero
    quotes: string[];             // Citas
    footerText?: string;          // Texto del footer/disclaimer
  };

  // ── Estructura usada ──
  blockSequence: string[];        // ['header','hero','text','bullets','cta','footer']
  templateIdUsed?: string;        // ID del template base (si se usó uno)

  // ── Contexto de la generación (solo si source !== 'manual') ──
  aiContext?: {
    userPrompt: string;           // Prompt que escribió el usuario
    reasoning?: string;           // Explicación de la IA
    tone?: string;                // Tono usado
    length?: string;              // Longitud pedida
  };

  // ── Referencia al proyecto ──
  mailingProjectId: string;       // ID del mailingProject guardado

  // ── Metadata ──
  createdBy: string;
  createdAt: Timestamp;
}
```

### 2.3 Por qué esta estructura

| Decisión | Motivo |
|----------|--------|
| **Textos desglosados** (no bloques crudos) | La IA necesita textos limpios para referenciar, no JSON de bloques con estilos |
| **`emailType` obligatorio** | Permite filtrar: "dame solo los textos científicos de esta marca" |
| **`source` tripartito** | Distingue manual / AI puro / AI editado. Los `ai_edited` son los más valiosos (el cliente validó el contenido) |
| **`tags` libres** | Para buscar por tema: 'eficacia', 'seguridad', 'webinar', 'congreso' |
| **1 doc por mailing** (no por texto) | Mantiene el contexto de cada email como unidad, permite ver la estructura completa |

### 2.4 Cuándo se almacena

```
┌──────────────────────────────────────────────────────────────┐
│                    FLUJO DE GUARDADO                          │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  MANUAL:                                                     │
│  Usuario crea email a mano → Guarda → Se extrae textos      │
│  de los bloques → Se pide emailType si no se seleccionó →   │
│  Se guarda en brandTextBank con source='manual'              │
│                                                              │
│  AI SIN EDITAR:                                              │
│  Usuario genera con AI → Guarda sin cambiar nada →           │
│  Se guarda con source='ai'                                   │
│                                                              │
│  AI EDITADO:                                                 │
│  Usuario genera con AI → Edita bloques → Guarda →            │
│  Se guarda con source='ai_edited' (los textos finales,       │
│  no los originales de la IA)                                 │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

**Momento exacto**: Al hacer `handleSave()` en MailingEditor.tsx, después de guardar el `mailingProject`, se llama a `saveToBrandTextBank()` que:
1. Extrae los textos de los bloques
2. Clasifica el `source` (manual / ai / ai_edited)
3. Pide `emailType` si no se asignó durante la generación AI
4. Guarda el documento en `brandTextBank`

### 2.5 Extracción de textos desde bloques

```typescript
function extractTextsFromBlocks(blocks: MailingBlockContent[]): BrandTextBankEntry['texts'] {
  const texts = {
    titles: [] as string[],
    paragraphs: [] as string[],
    bulletPoints: [] as string[],
    ctaTexts: [] as string[],
    ctaLabels: [] as string[],
    heroTitles: [] as string[],
    heroSubtitles: [] as string[],
    quotes: [] as string[],
    footerText: undefined as string | undefined,
  };

  for (const block of blocks) {
    switch (block.type) {
      case 'text':
        if (block.style?.headingLevel || block.style?.fontWeight === 'bold') {
          texts.titles.push(block.content);
        } else {
          texts.paragraphs.push(block.content);
        }
        break;
      case 'hero':
        if (block.style?.heroTitle) texts.heroTitles.push(block.style.heroTitle);
        if (block.style?.heroSubtitle) texts.heroSubtitles.push(block.style.heroSubtitle);
        break;
      case 'bullets':
        texts.bulletPoints.push(...block.content.split('\n').filter(Boolean));
        break;
      case 'cta':
        if (block.ctaText) texts.ctaTexts.push(block.ctaText);
        if (block.content) texts.ctaLabels.push(block.content);
        break;
      case 'quote':
        texts.quotes.push(block.content);
        break;
      case 'footer':
        texts.footerText = block.content;
        break;
    }
  }

  return texts;
}
```

## 3. SMART PROMPT: Generación inteligente del texto "Describe tu email"

### 3.1 UX del botón

```
┌──────────────────────────────────────────┐
│  Describe tu email                       │
│  ┌────────────────────────────────────┐  │
│  │                                    │  │
│  │  (textarea vacío o con texto)      │  │
│  │                                    │  │
│  └────────────────────────────────────┘  │
│  127 caracteres          ✨ Sugerir prompt│
└──────────────────────────────────────────┘
```

- **Botón**: "✨ Sugerir prompt" — alineado a la derecha, debajo del textarea
- **Comportamiento**: 
  - Si el textarea está vacío → genera un prompt completo basado en marca + banco de textos
  - Si el textarea tiene texto → mejora/expande el texto existente
- **Loading**: El botón muestra un spinner mientras genera (1-3 segundos)
- **Resultado**: Reemplaza el contenido del textarea con el prompt generado

### 3.2 Lógica de generación del Smart Prompt

El Smart Prompt consulta el banco de textos y genera un brief que sea **coherente con lo que la marca ya ha producido**:

```typescript
async function generateSmartPrompt(

  tenantId: string,
  emailType?: EmailType,
  currentText?: string,

```

**System prompt para Smart Prompt:**

```

  - Títulos usados: "Eficacia comprobada", "Resultados del estudio", ...
  - CTAs: "Ver estudio", "Inscríbete", ...

Tipo: Científico (3 emails)
  - Subjects: "Publicación del ensayo...", ...
  - Títulos usados: "Resultados fase III", ...

PATRONES DETECTADOS:
- Tipo más frecuente: Promocional (60%)
- Tono habitual: profesional
- Temas recurrentes: eficacia, seguridad, webinar
- Largo típico de subjects: 35-50 caracteres

{SI currentText no está vacío}:
El usuario ya escribió esto: "{currentText}"
Mejóralo y expándelo manteniendo el estilo de la marca.
{SINO}:
Genera un prompt de email que sea coherente con el estilo y temas
habituales de esta marca. {SI emailType}: Debe ser de tipo {emailType}.

{FIN SI}

REGLAS:
- Máximo 2-3 oraciones
- Ser específico (mencionar molécula, indicación, datos concretos)
- Incluir contexto de audiencia (ej: "dirigido a cardiólogos")
- Mantener el tono habitual de la marca

```

### 3.3 Flujo completo

```
1. Usuario abre panel "Crear con AI"
2. Selecciona tipo de email (opcional)
3. Click "✨ Sugerir prompt"
4. Sistema:
   a. Consulta brandTextBank (últimas 15 entries de esta marca)
   b. Agrupa por emailType, extrae patrones
   c. Llama a Gemini con brand context + banco de textos + emailType
   d. Recibe texto de prompt
5. Se llena el textarea con el prompt sugerido
6. Usuario lo edita si quiere, luego hace click "Generar Email"
```

## 4. CORPUS-BASED GENERATION: La IA genera basándose en el banco

### 4.1 Diferencia fundamental

**Antes**: La IA genera un email "de cero" usando solo claims + insights + brand colors.  
**Ahora**: La IA genera un email **referenciándose en los textos anteriores de la marca**.

Esto significa que:
- Si la marca suele usar títulos cortos y directos → la IA los genera así
- Si siempre incluyen bullets con datos porcentuales → la IA mantiene ese patrón
- Si el tono es siempre académico → la IA no genera algo "cercano"
- Si nunca usan hero image → la IA no la incluye

### 4.2 Cambio en `AIMailingContext`

Se agrega un nuevo campo al contexto:

```typescript
// Agregar a AIMailingContext en aiMailingContext.ts
export interface AIMailingContext {
  // ... campos existentes ...

  // ── Banco de textos de esta marca ──
  textBank?: {
    totalEmails: number;
    byType: Record<string, number>;     // { 'promocional': 5, 'científico': 3 }

    // Textos agrupados para referencia directa de la IA
    recentSubjects: string[];           // Últimos 10 subjects
    recentTitles: string[];             // Últimos 15 títulos
    recentParagraphs: string[];         // Últimos 10 párrafos (recortados a 100 chars)
    recentBulletPoints: string[];       // Últimos 15 bullet points
    recentCTAs: string[];              // Últimos 10 CTAs
    recentHeroTitles: string[];        // Últimos 5 hero titles

    // Análisis de patrones
    preferredTone?: string;
    preferredLength?: string;
    avgSubjectLength: number;
    commonBlockSequences: string[];     // ['header→hero→text→cta→footer', ...]

    // Resumen textual para el system prompt
    summary: string;
  };
}
```

### 4.3 Inyección en el System Prompt

En `buildMailingSystemPrompt()` de `aiMailingGenerator.ts`:

```
{SI context.textBank Y textBank.totalEmails > 0}:

═══════════════════════════════════════════
BANCO DE TEXTOS DE ESTA MARCA ({textBank.totalEmails} emails anteriores)
═══════════════════════════════════════════

DISTRIBUCIÓN POR TIPO:
{Object.entries(textBank.byType).map(([type, count]) => `- ${type}: ${count} emails`)}

SUBJECTS ANTERIORES (REFERENCIA DE ESTILO):
{textBank.recentSubjects.map((s, i) => `${i+1}. "${s}"`)}

TÍTULOS USADOS:
{textBank.recentTitles.map(t => `- ${t}`)}

BULLET POINTS DE REFERENCIA:
{textBank.recentBulletPoints.map(b => `- ${b}`)}

CTAs HABITUALES:
{textBank.recentCTAs.map(c => `- ${c}`)}

ANÁLISIS DE ESTILO:
{textBank.summary}

INSTRUCCIONES BASADAS EN EL BANCO DE TEXTOS:
- GENERA textos con el MISMO estilo, tono y vocabulario que los ejemplos anteriores
- Mantén la misma longitud promedio de subjects ({textBank.avgSubjectLength} chars)
- Usa estructuras de bloque similares: {textBank.commonBlockSequences[0]}
- NO copies literalmente textos anteriores, pero mantén el mismo nivel de formalidad
- Si el tipo de email solicitado tiene ejemplos anteriores, úsalos como referencia principal
- Si NO tiene ejemplos de este tipo, adapta el estilo general de la marca al nuevo tipo

{FIN SI}
```

### 4.4 Filtrado inteligente por tipo de email

Cuando el usuario selecciona un tipo de email (ej: "Científico"), el banco se filtra:

```typescript
// En buildAIMailingContext(), al construir el textBank:
const allEntries = await getBrandTextBank(brandId, tenantId, { limit: 30 });

// Si hay emailType seleccionado, priorizar entries de ese tipo
const sameTypeEntries = emailType
  ? allEntries.filter(e => e.emailType === emailType)
  : [];
const otherEntries = emailType
  ? allEntries.filter(e => e.emailType !== emailType)
  : allEntries;

// Combinar: primero del mismo tipo, luego otros (para dar contexto general)
const prioritizedEntries = [...sameTypeEntries.slice(0, 10), ...otherEntries.slice(0, 5)];
```

Así la IA recibe primero los textos más relevantes (mismo tipo) y luego contexto general del estilo de la marca.

## 5. SELECTOR DE TIPO AL GUARDAR (emails manuales)

### 5.1 El problema

Cuando el usuario crea un email manualmente (sin AI), no hay `emailType` asignado. Necesitamos que al guardar por primera vez, se le pregunte qué tipo de email es.

### 5.2 UX

Al hacer click en "Guardar" por primera vez (crear nuevo proyecto), si no hay `emailType`:

```
┌─────────────────────────────────────┐
│  ¿Qué tipo de email es este?        │
│                                     │
│  [Promocional] [Informativo]        │
│  [Newsletter]  [Invitación]         │
│  [Científico]  [Aviso breve]        │
│  [Otro]                             │
│                                     │
│  (Esto nos ayuda a mejorar las      │
│   sugerencias de IA para tu marca)  │
│                                     │
│        [Guardar]  [Omitir]          │
└─────────────────────────────────────┘
```

- **Si selecciona tipo**: Se guarda con ese tipo
- **Si omite**: Se guarda con tipo `'otro'`
- **Emails AI**: Ya tienen tipo asignado desde el panel "Crear con AI", no se pregunta

### 7.2 `buildTextBankSummary()` — Análisis de patrones

```typescript
function buildTextBankSummary(entries: BrandTextBankEntry[]): AIMailingContext['textBank'] {
  // Conteo por tipo
  const byType: Record<string, number> = {};
  entries.forEach(e => {
    byType[e.emailType] = (byType[e.emailType] || 0) + 1;
  });

  // Extraer textos recientes
  const recentSubjects = entries.slice(0, 10).map(e => e.subject);
  const recentTitles = entries.flatMap(e => e.texts.titles).slice(0, 15);
  const recentParagraphs = entries.flatMap(e => e.texts.paragraphs)
    .map(p => p.slice(0, 100)).slice(0, 10);
  const recentBulletPoints = entries.flatMap(e => e.texts.bulletPoints).slice(0, 15);
  const recentCTAs = entries.flatMap(e => e.texts.ctaTexts).slice(0, 10);
  const recentHeroTitles = entries.flatMap(e => e.texts.heroTitles).slice(0, 5);

  // Tono preferido (de entries con aiContext)
  const tones = entries.filter(e => e.aiContext?.tone).map(e => e.aiContext!.tone!);
  const preferredTone = mode(tones);

  // Largo promedio de subjects
  const avgSubjectLength = Math.round(
    recentSubjects.reduce((sum, s) => sum + s.length, 0) / (recentSubjects.length || 1)
  );

  // Secuencias de bloques comunes
  const sequences = entries.map(e => e.blockSequence.join('→'));
  const commonBlockSequences = [...new Set(sequences)].slice(0, 3);

  // Resumen textual
  const topType = Object.entries(byType).sort((a, b) => b[1] - a[1])[0];
  const summary = [
    `La marca ha producido ${entries.length} emails.`,
    topType ? `El tipo más frecuente es "${topType[0]}" (${topType[1]} emails).` : '',
    preferredTone ? `El tono habitual es ${preferredTone}.` : '',
    `Largo promedio de subjects: ${avgSubjectLength} caracteres.`,
    recentCTAs.length > 0 ? `CTAs típicos: "${recentCTAs.slice(0, 3).join('", "')}"` : '',
  ].filter(Boolean).join(' ');

  return {
    totalEmails: entries.length,
    byType,
    recentSubjects,
    recentTitles,
    recentParagraphs,
    recentBulletPoints,
    recentCTAs,
    recentHeroTitles,
    preferredTone,
    preferredLength: undefined, // se infiere del avgSubjectLength
    avgSubjectLength,
    commonBlockSequences,
    summary,
  };
}
```

## 8. SEGURIDAD Y PERFORMANCE

### 8.1 Límites

| Aspecto | Límite | Motivo |
|---------|--------|--------|
| Entries por query | Máximo 30 últimas | Performance de lectura + tokens |
| Textos en prompt | Máximo ~2000 tokens del banco | No sobrecargar el system prompt |
| Summary length | Máximo 500 chars | Concisión |
| Smart prompt debounce | 1 cada 5 segundos | Rate limit Gemini |
| Entries totales | Sin límite (1 doc por mailing guardado) | Firestore escala bien |

### 8.2 Privacidad

- Filtrado por `tenantId` (multi-tenant)
- El banco de textos solo se usa para la misma marca del mismo tenant
- Los textos del cliente nunca se comparten entre tenants
- Futuro: botón "Borrar banco de textos" por marca

### 8.3 Costo adicional

| Operación | Costo estimado |
|-----------|---------------|
| Smart prompt (Gemini) | ~$0.0005 por sugerencia |
| Query 30 docs Firebase | ~$0.0015 (30 reads) |
| Write 1 entry al guardar | ~$0.0001 (1 write) |
| **Total extra por generación** | ~$0.002 |

## 10. RESUMEN

### Lo que se agrega:

1. **Colección `brandTextBank`** — Almacena TODOS los textos de cada marca (manuales + AI). Cada entrada taggeada por tipo (informativo, científico, etc.) y fuente (manual / ai / ai_edited).
2. **Corpus-Based Generation** — La IA genera emails **referenciándose en textos anteriores** de la marca. No empieza de cero: conoce el estilo, tono, vocabulario y estructura del cliente.
3. **"✨ Sugerir prompt"** — Genera briefs inteligentes usando el banco completo de textos como referencia.

### Beneficio progresivo:
```
Email 1:   IA solo conoce claims + insights + colores (genérico)
Email 5:   IA tiene 5 emails de referencia, detecta patrón de estilo
Email 20:  IA genera subjects, títulos y CTAs en el MISMO estilo del cliente
Email 50:  Smart Prompt sugiere temas nuevos complementarios a lo ya creado
```

### Diferencia clave vs plan anterior:
- **Antes**: Solo se almacenaban interacciones AI → la IA aprendía solo de lo que ella generaba
- **Ahora**: Se almacena TODO (manual + AI) → la IA aprende del **estilo real del cliente**, no solo de sus propias generaciones
